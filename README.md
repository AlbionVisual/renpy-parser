# Ren'Py Parser

Проект предназначен для автоматизированного извлечения диалогов, переводов и аудио-контекста (музыки и звуковых эффектов) из игр на движке Ren'Py. Главная цель — получить структурированные табличные данные из сырых файлов игры. Это необходимо для создания датасетов для машинного обучения, анализа качества переводов, поиска по тексту игры или создания модов.

Суть работы парсера заключается в симуляции (трассировке) выполнения скриптов игры. Скрипт не просто выдёргивает все строки подряд, а пытается восстановить логический порядок исполнения `label`‑ов, начиная с `label start`. Попутно он отслеживает команды включения и выключения музыки (`play music`, `stop music`) и звуков (`play sound`), чтобы для каждой реплики персонажа точно знать, какой трек играл на фоне и какой звуковой эффект ей предшествовал.

Весь процесс полностью автоматизирован: от поиска zip-архивов с играми в папке загрузок до декомпиляции скриптов, парсинга и, при желании пользователя, загрузки итогового датасета напрямую в базу данных PostgreSQL.

## Структура проекта

Проект организован вокруг нескольких ключевых директорий:

- В корне лежит [`.gitignore`](.gitignore): по умолчанию не коммитятся тяжёлые/генерируемые каталоги (`audios/`, `output/`, `raw-games/`, артефакты Python, настройки Cursor и т.д.).

- `**Downloads/**` (папка загрузок системы, по умолчанию в Windows) — отсюда скрипт по умолчанию забирает исходные архивы с играми (`.zip`, `.rar`, `.7z`).
- `**.unpack_work/**` — временная директория, куда архивы копируются перед распаковкой (особенно актуально при работе из-под WSL).
- `**raw-games/**` — сюда складываются распакованные игры. После очистки от мусора здесь остаются только скрипты (`.rpy`), папки с переводами (`tl/`) и аудиофайлы.
- `**output/**` — сюда сохраняются промежуточные результаты парсинга в формате `.csv` (по одному файлу на каждую игру).
- `**tools/**` — папка для сторонних утилит. В частности, здесь должен находиться декомпилятор `unrpyc`.

## Используемые инструменты и технологии

- **Python 3** — основной язык разработки пайплайна.
- **PostgreSQL** — целевая база данных для хранения итоговых датасетов.
- **psycopg3** (`psycopg`) — современный драйвер для быстрой потоковой записи данных в PostgreSQL.
- **unrpa** — утилита для распаковки `.rpa` архивов Ren'Py.
- **unrpyc** — декомпилятор скомпилированных скриптов (`.rpyc` -> `.rpy`).
- **7z / unrar** — системные утилиты для извлечения исходных архивов игр.

**Схема пайплайна:**
`[ZIP/RAR в Downloads] ➔ (unpack_archives) ➔ [raw-games] ➔ (unpack_renpy + unrpa/unrpyc) ➔ [Декомпилированные .rpy] ➔ (parse_rpy) ➔ [CSV в output] ➔ (Интерактивное подтверждение) ➔ [PostgreSQL] ➔ (music_scripts/text_scripts: постобработка для ML) ➔ [Готовые фичи/тексты для моделей]`

## Инструкция по использованию

### Подготовка

Перед началом работы убедитесь, что у вас установлен PostgreSQL. В корне репозитория скопируйте [`.env.template`](.env.template) в `.env` и заполните переменную `pghost` (DSN для `psycopg3`). Файл `.env` не коммитится (см. [`.gitignore`](.gitignore)).

```env
pghost=postgresql://user:password@localhost:5432/dbname
```

### Запуск

Основной запуск пайплайна выполняется через `main.py`. По умолчанию скрипт ищет архивы в папке загрузок Windows и копирует их в WSL.

```bash
python3 main.py
```

После того как базовый датасет загружен в PostgreSQL, обычно выполняют постобработку:
- заполнение/нормализация справочника треков `music_data` и привязка музыки к репликам
- формирование и очистка единой текстовой колонки `text1` под модели (эмбеддинги/TF‑IDF и т.п.)

**Доступные флаги:**

- `--no-copy` — не копировать архив во временную папку WSL, а читать напрямую из папки загрузок (экономит место, но может быть медленнее).
- `--downloads <путь>` — указать кастомный путь к папке с архивами игр.
- `--no-file-filter` — отключить удаление "лишних" файлов из папки `game/` после распаковки.

### Пайплайн работы (Интерактивный режим)

1. Вы запускаете скрипт. Он находит архивы, распаковывает их, извлекает `.rpa` и декомпилирует `.rpyc`.
2. Начинается парсинг. Скрипт обрабатывает **все** найденные игры и сохраняет результаты в папку `output/` в виде `.csv` файлов.
3. После того как все CSV сгенерированы, начинается этап сохранения в БД.
4. Скрипт по очереди спрашивает про каждую игру:
  `Игра <имя>: записать в БД и удалить временные файлы [Y/n]?`
5. В этот момент выполнение приостанавливается. Вы можете открыть сгенерированный `.csv` файл в папке `output/` и проверить корректность данных.
6. Если всё хорошо, вы нажимаете `Enter` (или вводите `y`). Скрипт:
  - Моментально загружает данные в PostgreSQL.
  - Удаляет исходный архив из папки загрузок.
  - Удаляет распакованную папку игры из `raw-games/`.
  - Удаляет просмотренный `.csv` файл.
7. Если вы вводите `n`, скрипт пропускает эту игру (ничего не пишет в БД и ничего не удаляет).
8. (Опционально) Постобработка в БД для ML:
  - `music_scripts/parse_folder_to_db.py <папка_с_аудио>` — заполнить `music_data(game, path)` из локальных файлов
  - `text_scripts/fill_text1_interactive.py` — выбрать для каждой игры источник (`text_orig` или `text_translated`) и заполнить `text1`
  - `text_scripts/clean_text1.py` — очистить/нормализовать `text1` (плейсхолдеры, теги, мусорные символы и т.д.)

### Проверка в базе данных

Чтобы убедиться, что данные успешно загрузились, зайдите в консоль PostgreSQL (`psql`) и выполните проверочные запросы:

```sql
-- Проверить общее количество загруженных строк:
select count(*) from text_musics_rel;

-- Посмотреть распределение строк по играм:
select game, count(*) from text_musics_rel group by game;
```

Для быстрых ручных проверок/тестов “напрямую в БД” есть готовые подборки запросов:
- `music_scripts/db_updaters.md` — заметки и SQL для апдейтов/нормализации музыки
- `text_scripts/sql_queries.md` — заметки и SQL для проверки/поиска проблем в тексте и `text1`

## Справочник по коду (для AI-ассистентов и разработчиков)

Ниже представлена подробная техническая "карта" кодовой базы. Она предназначена для быстрого погружения в архитектуру, понимания потоков данных (data flow), структур состояния и эвристик парсинга.

### `main.py`

Оркестратор пайплайна.

- Парсит аргументы командной строки (`--no-copy`, `--downloads`, `--no-file-filter`).
- Последовательно вызывает: `unpack_archives.main` ➔ `unpack_renpy.main` ➔ `parse_rpy.main`.
- **Важно:** Передаёт `archive_mapping` (словарь: `имя_папки_игры` -> `путь_к_исходному_zip`) из первого этапа в последний, чтобы `parse_rpy` знал, какой архив удалять при подтверждении пользователем.

### `consts.py`

Файл конфигурации и констант.

- **Директории:** `BASE_DIR`, `RAW_GAMES`, `WORK_DIR` (для WSL), `OUTPUT_DIR`, `AUDIOS_DIR`.
- **Схема БД:** `TABLE_NAME`, `CREATE_TABLE_SQL`.
- **Маппинг данных:** `FIELDNAMES` (словарь, связывающий человекочитаемые ключи с именами колонок в БД/CSV) и `FIELDNAMESINORDER` (строгий порядок колонок для CSV).
- **Эвристики парсинга:** `SKIP_FILES` (игнорируемые файлы, например `screens.rpy`), `STMT_KEYWORDS` (ключевые слова Ren'Py DSL, используемые для фильтрации служебных строк от реальных реплик), `BASE_INDENT` (отступ, используемый для определения вложенности блоков кода).

### `csv_sync_db.py`

Модуль взаимодействия с PostgreSQL (использует `psycopg3`).

- `db_create_table()` / `check_table()`: DDL операции.
- `db_add_dataset(dataset: list[dict])`: **Критичная для производительности функция.** Вместо медленных `INSERT` в цикле, использует `COPY {TABLE_NAME} FROM STDIN` через `cur.copy()` и `copy.write_row()`. Это позволяет загружать тысячи строк за миллисекунды в рамках одной транзакции.

### `unpack_archives.py`

Модуль поиска и первичной распаковки архивов (`.zip`, `.7z`, `.rar`).

- `need_extract(name, no_file_filter)`: Логика фильтрации "на лету". Пропускает только файлы внутри папки `game/`, оставляя `.rpa`, `script.rpy`, папки `tl/russian`, `tl/ru` и `audio/`.
- `first_folder_from_names(names)`: Эвристика для определения корневой папки игры внутри архива (игнорирует служебные папки и папки, состоящие только из цифр).
- **Возврат:** Возвращает код успешности и `archive_mapping` для последующего удаления мусора.

### `unpack_renpy.py`

Модуль подготовки ресурсов Ren'Py.

- `check_tools()`: Валидация зависимостей (`unrpa` в PATH/local, `unrpyc` в `tools/`).
- `process_game_folder()`: Выполняет работу в два этапа с промежуточными очистками:
  1. Распаковка `.rpa` ➔ удаление `.rpa` ➔ очистка мусора (оставляем только `.rpy`, `.rpyc`, `tl/`, аудио).
  2. Декомпиляция `.rpyc` в `.rpy` ➔ финальная очистка (удаление `.rpyc`).
- `move_audio_files()`: Собирает все `.ogg`, `.mp3`, `.wav` файлы из подпапок и копирует их в централизованную директорию `AUDIOS_DIR` (если это требуется логикой).

### `scan_rpy_dir.py`

Модуль статического анализа (AST-подобный сбор метаданных без исполнения).

- `scan_labels()`: Ищет все `label <name>:` через регулярные выражения, сохраняет маппинг `label -> {file, line}`.
- `scan_defines()`: Собирает переменные и алиасы аудио (`define audio.X = "path"` или `$ var = "path"`). Формирует `audio_map` и `all_vars`.
- `scan_translations()`: Парсит файлы локализации в `tl/russian`. Извлекает переведённые строки, привязывая их к ключу `(label_name, replica_index)`. Использует `_extract_label_from_id` для парсинга хешей Ren'Py.

### `trace_labels.py`

**Ядро парсера.** Модуль динамической трассировки (стейт-машина).

- `build_execution_order()`: Читает файлы в память. Склеивает многострочные строки. Вычисляет границы каждого `label` (от строки объявления до следующего `label` или конца файла). Вырезает блоки `screen ...`, так как они не относятся к линейному сюжету.
- `extract_dialogue_text(line)`: Эвристический парсер реплик. Отбрасывает комментарии, Python-код (`$`), меню (`"..." :`). Разделяет строку на `speaker` (если есть) и `text`.
- `resolve_audio()`: Рекурсивно разыменовывает переменные аудио, используя `audio_map` и `all_vars`, пока не получит реальный путь к файлу или не выдаст `UNKNOWN:var`.
- `trace_game()`: Запускает обход графа выполнения, начиная с `label start`.
  - **Стейт-машина:** Хранит текущее состояние в словаре `state` (`music`, `music_loop`, `pending_sfx`, `sfx_invalidated`).
  - **Обработка аудио:** При встрече `play music` обновляет стейт. При `play sound` записывает `pending_sfx`. При `pause` инвалидирует `pending_sfx` (звук проиграл до реплики).
  - **Управление потоком:** Обрабатывает `jump` (переход), `call` (вложенный вызов) и `return` (возврат), опираясь на отступы (`indent > BASE_INDENT`) для определения вложенности блоков.
  - **Сборка датасета:** При нахождении реплики формирует словарь со всеми текущими стейтами (музыка, звук, файл, строка) и подтягивает перевод из `tl_map`.

### `parse_rpy.py`

Модуль интеграции и пользовательского взаимодействия.

- `process_game()`: Вызывает сканирование и трассировку для одной игры. Возвращает готовый `dataset` (list of dicts) и путь к сохранённому `.csv`.
- `main()`: 
  - **Pass 1 (Пакетная обработка):** В цикле парсит все игры из `raw-games/`, генерируя `.csv` файлы в `output/`. Сохраняет результаты в список `processed_games`.
  - **Pass 2 (Интерактивный CLI):** В цикле проходится по `processed_games` и запрашивает у пользователя подтверждение (`input()`).
  - **Cleanup:** При подтверждении (`Y`) вызывает `db_add_dataset`, затем удаляет: исходный ZIP (через `archive_mapping`), папку игры (через `shutil.rmtree`) и временный `.csv` (через `unlink`).

### `music_scripts/`

Постобработка аудио-данных и привязка музыки к справочнику `music_data`.

- `parse_folder_to_db.py`: Сканирует папки с аудио по играм и заполняет `music_data(game, path)`.
- (DB updater-скрипты): Нормализуют `music_path` и/или заполняют `text_musics_rel.music` (foreign key на `music_data.id`) с помощью эвристик сопоставления путей (точное совпадение, подстрока, удаление расширений, обработка `UNKNOWN:` и т.д.).

### `text_scripts/`

Постобработка текста: формирование единой колонки под эмбеддинги/модели и её очистка.

- `fill_text1_interactive.py`: Для каждой игры интерактивно выбирает источник текста (`text_orig` или `text_translated`) и заполняет `text1`.
- `clean_text1.py`: Последовательная нормализация `text1` (схлопывание пробелов/многоточий, обработка плейсхолдеров, снятие ремарок в `[...]`, удаление Ren'Py-тегов `{...}`, мягкая фильтрация мусорных символов и т.п.). Строки, которые после нормализации стали пустыми/нецелевыми, можно исключать при обучении.

### ML: RuBERT → музыка (`text_scripts/ml_scripts/`)

Скрипты `train_rubert_to_music_*.py` при `--split leave_games_out` поддерживают отбор **только test-игр** по числу строк в загруженном датасете (чтобы мелкие игры не попадали в test):

`--held-out-game-min-rows 1000 --held-out-game-max-rows 5000`

Нужно задавать **оба** порога; с `--split within_game` эти флаги недопустимы. Если игр в диапазоне меньше, чем `--test-games`, запуск завершится с ошибкой.

### Стиль кода и материалы от ИИ

Чтобы репозиторий не превращался в «типичный ИИ-слайм», имеет смысл заранее зафиксировать правила в [правилах Cursor](https://docs.cursor.com/context/rules-for-ai) или `AGENTS.md`: короткие имена без лишних префиксов, не раздувать публичные докстринги, не дублировать очевидные типы в каждой мелкой функции, не оставлять декоративные комментарии «что делает следующая строка». Новый код лучше подгонять под стиль соседних файлов, а не под шаблон чата.
