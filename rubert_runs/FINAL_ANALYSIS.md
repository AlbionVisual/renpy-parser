Протокол сравнения: **только `top_percent = 35`**, сплит **`leave_games_out`**, **`held_out_games_seed = 17`**, **`seed = 40`** (где не указано иное — см. отдельные раны). Метрика для рейтинга: **cosine hit rate** из `metrics.json` → `retrieval.cosine` (`hits / totals`).

## 1. Что не смешивать с основной линией

- **`run_020_top_percent_50_diag`**: `top_percent=50` — на диске cos-hit **~0.705**, но это **другая постановка** (легче попасть в top 50% кандидатов). В тексте как «лучшая модель» **не использовать**.
- **`run_002_gru`**: `top_percent=15`, другие сиды — только как отдельное упоминание.
- **`local_lstm_baseline` / `local_lstm_masked`**: короткие прогоны (**5 эпох**), не сопоставимы с полными 40–100+ эпохами; cos-hit **~0.555** совпадает с плато нескольких полных LSTM — это **эмпирическое совпадение**, не доказательство эквивалентности обучения.

## 3. Импакт столбцов по `per_column_metrics.csv` (test)

Файлы CSV в папке рана относятся к **test**; маска retrieval по умолчанию считает пороги на **val**. Для подбора фильтра CSV — **ориентир**, после прогона смотреть warning с квантилями в логе.

Скрипт сводки (квантили, число столбцов с `pearson >= t`, медиана |ρ| по префиксам):

```bash
python3 text_scripts/ml_scripts/summarize_per_column_csv.py rubert_runs/run_002_lstm/per_column_metrics.csv
python3 text_scripts/ml_scripts/summarize_per_column_csv.py rubert_runs/run_019_autoreg_light_budget/per_column_metrics.csv
```

### 3.1. Семейство LSTM (`run_002_lstm`)

По группам имён столбцов (median |ρ| на test):

- **mfcc** — **0.41** (сильнее всего; в топе по |ρ| — mfcc_*, **tempo**)
- spectral, chroma, jamendo — слабее по медиане, чем mfcc

То есть для **LSTM** «информативные» для предсказания оси — прежде всего **MFCC и темп**, а не jamendo-теги.

### 3.2. Лёгкая авторегрессия (`run_019_autoreg_light_budget`)

- **jamendo** — median |ρ| **~0.35**, **chroma** **~0.37**
- **mfcc** — median |ρ| **~0.08** (модель почти не «вытягивает» MFCC относительно тегов)

Иначе говоря, **autoreg light опирается на jamendo/chroma**, классический **LSTM — на mfcc/tempo**; один и тот же числовой фильтр по `pearson` **нельзя** считать универсальным.

## 4. Рекомендации по `--retrieval-column-filter`

Жёсткий фильтр (в духе `mae<0.55` и узкий `pearson`) легко **убирает оси, важные для cosine retrieval**, поэтому маска может **удушить hit** даже при «хорошей» регрессии по выбранным столбцам — это видно на локальном `local_lstm_masked` vs baseline.

**Мягкие стартовые точки (после проверки на val в логах):**

- Для **LSTM** (семантика «mfcc сильные»):  
  `pearson>=8&&mae<0.75`  
  (в парсере `pearson>=8` → **0.08**; `mae` — в исходных единицах.)

- Для **autoreg** (семантика «jamendo/chroma сильные»):  
  `pearson>=10&&mae<0.85`  
  (`pearson>=10` → **0.10**.)

В обоих случаях оставить **`--retrieval-mask-min-columns 10`** и **`--retrieval-mask-fallback-percentile 75`**, чтобы не получить пустой набор и не полагаться только на ручной порог.

Готовые однострочники — в **`rubert_runs/manifest_final_reruns.jsonl`** (строки `final_masked_lstm_soft` и `final_masked_autoreg_soft`).

## 5. Ваш случай «57 → 43»

Число **~0.555** для короткого LSTM **совпадает** с плато многих полных прогонов @35% — это не 0.57 как отдельный режим, а **округление / совпадение шкалы**. Падение **до ~0.44** при агрессивной маске согласуется с тем, что **retrieval — другая задача**, чем «хороший MAE по подмножеству столбцов**; окончательно подтверждается только сравнением двух прогонов с одним чекпоинтом и разной маской.
