# RuBERT→music: сводка скачанных прогонов

Все конфигурации при `leave_games_out`, `test_games=3`, **`held_out_games_seed=17`** (кроме **run_002_gru**, где **`held_out_games_seed=42`** — результаты **нельзя** ставить в один ряд с остальными без оговорки).

**Валидация:** `--val-split tail`, доля строк в конце **внутри тех же игр**, что train. **Тест:** выбранные **целиком другие игры**. Поэтому **val Pearson/MAE часто заметно лучше**, чем test — это ожидаемый разрыв *same-game tail* vs *unseen games*, а не признак «идеальной» модели.

**Retrieval:** для каждой тестовой строки ранжируются **все треки той же игры-кандидаты** по близости предсказанного вектора; hit, если истинный `music_id` в топ **ceil(top_percent × N/100)**. Для большинства прогонов **`top_percent=35`** ⇒ случайный baseline ≈ **35%**. Исключение: **run_002_gru** записан с **`top_percent=15`** ⇒ baseline ≈ **15%**.

## Выводы по архитектурам и «усложнению»

1. **Крупные трансформеры** (особенно большие `embedding-lags` наряду с уже тяжёлой последовательной моделью) на **test** показывают **плоский или случайный** Pearson при по-прежнему высоком **cosine между векторами** (эмбеддинговая траектория в пространстве таргета), но **ретривал-хит** может просесть относительно более простых моделей — типичное **несоответствие** «угла вектора» и «выбора id трека внутри пула игры».

2. **Очень широкий MLP с lag** (run_008) даёт высокое **соответствие на val** но **переносится хуже** — похоже на заучивание внутренних паттернов **по уже виденным играм** с хвостом val.

3. **LSTM поменьше lag (4)** + умеренный размер (run_002) — наиболее устойчивый текущий **баланс** для **hit@35 (cos)**.

4. **Autoreg**: лучше объясняет таргеты на test по Pearson, но тормозит; имеет смысл **дорогие** прогоны с **минимальными** изменениями и лимитированным перебором.

5. Цель «**hit ≈ 90%** при том же **`top_percent=35`**»: это **разрыв ~+30 пунктов** к лучшему текущему ~**60%**. Один только перебор гиперпараметров **мало вероятно** добьётся этого; вероятнее потребуется другая постановка (ранжирующая/контрастивная цель по трекам игры), больше данных или другая разметка. Тем не менее перебор вокруг **run_002** и **компактный MLP/TCN** рационален.

## Про cosine-hit vs RMSE-hit

Подбор лучших кандидатов по косинусу и по среднеквадратической ошибке **не эквивалентен** (`retrieval_scores` в коде). Поэтому нормален один прогон, где один скоринг лучше другого (**run_010** vs **run_002** по разным осям).
